# TTRPG Tools — платформа инструментов для настольных ролёвок

**Статус:** черновик требований (первая итерация)
**Дата:** 2026-06-17
**Источник вдохновения:** https://losing.games/2019-07-24-mausritter-locations/

## Цель

Сделать SPA-платформу личных инструментов для ведения настольных ролевых игр, которая:

1. Запускается из браузера без бэкенда и хостится бесплатно (GitHub Pages).
2. Поддерживает разные TTRPG-системы как независимые модули данных.
3. Стартует с одного работающего инструмента — генератора локаций Mausritter на русском.
4. Архитектурно готова расти до бестиариев, трекеров состояния и интерактивных инструментов (гекс-карты), но эти возможности не строятся в первой итерации.

## Аудитория

Один пользователь — автор и его игровой стол. Не публичный продукт. Это позволяет:

- Не вкладываться в SEO, локализацию (только русский), регистрацию пользователей, синхронизацию между устройствами.
- Свободно ломать UX между итерациями.
- Хранить любое состояние в `localStorage`, если оно вообще понадобится.

## Первая итерация — `/mausritter/locations`

Перевод и реинтерпретация генератора локаций Mausritter ([losing.games](https://losing.games/2019-07-24-mausritter-locations/)) на русский.

### Что должно работать

- Вводный текст оригинала переведён на русский (одно описательное вступление о масштабе гекскраула 1 миля/гекс и о том, что в каждом гексе один заметный ориентир).
- Селектор биома: `Countryside / Forest / River / Human town` → русские названия.
- Кнопка **«Бросить локацию»** генерирует:
  - один Landmark (d20 по выбранному биому),
  - одну Location detail (d20 из общей таблицы), включая открытый вопрос в скобках.
- Каждую из двух частей результата можно перебросить отдельно без потери второй.
- Под результатом видны исходные таблицы как справка (для тех, кто хочет кидать реальные кубики).
- В футере — атрибуция: «Based on Mausritter by Isaac Williams. <ссылка на лицензию>».

### Чего не делаем в первой итерации

- Журнал брошенных локаций («список заполненного гекса»).
- Сохранение результата в `localStorage`.
- Шеринг по URL (генерация `?roll=1-4_2-15`).
- Тёмная тема, PWA-манифест, оффлайн-режим.
- Карточки врагов, бестиарий, гекс-карта.

## Стек

| Слой | Решение | Почему |
| --- | --- | --- |
| Сборка / роутинг | Astro (static output) | Статика по умолчанию; островки React только там, где нужен интерактив. Хорошо ложится на разнородный контент (генераторы, справочники, трекеры). |
| Интерактив | React 18 + TypeScript | Большая экосистема, типизация данных таблиц. |
| UI-компоненты | shadcn + baseui | Копируемые компоненты, исходник в репо. Базовые примитивы — baseui (предпочтение автора). На этапе реализации проверить, что выбранная конфигурация shadcn-реестра ставится поверх baseui без конфликтов с Astro + React islands. |
| Стили | Tailwind CSS | Tailwind как механика. Дизайн-токены (цвета, типографика, скругления, отступы) — отдельный артефакт, создаётся на этапе реализации: DESIGN.md по образцу [bergside/awesome-design-skills — skills/ant/DESIGN.md](https://github.com/bergside/awesome-design-skills/blob/main/skills/ant/DESIGN.md) + [skills/ant/SKILL.md](https://github.com/bergside/awesome-design-skills/blob/main/skills/ant/SKILL.md). Конкретный «диалект» (ant / другой из реестра) выбираем в ce-plan. |
| Броски | свой `lib/dice.ts` на `crypto.getRandomValues` + rejection sampling сейчас, [`rpg-dice-roller`](https://github.com/dice-roller/rpg-dice-roller) когда понадобится нотация | Для честного равномерного распределения целых в `[1, sides]`: генерим `Uint32`, отбрасываем значения из «хвоста» который не делится на `sides` нацело, иначе будет modulo-bias (особенно заметный на не-степени-двойки вроде d20). `Math.random() * 20 \| 0 + 1` дешёво, но привязан к качеству PRNG движка и теряет точность на крупных кубах. `rpg-dice-roller` подключаем когда появятся выражения (`1d6+1` для урона, спасброски и т. п.). |
| Контент таблиц | TypeScript-файлы (`as const`) | Типизация на границе данных и UI, ошибки в таблицах ловятся компилятором. |
| Тексты правил | MDX | Перевод и описание лежат рядом с инструментом, легко править. |
| Хостинг | Cloudflare Pages | Статика бесплатно, отдельный домен/субдомен из коробки, **и есть Workers для серверных функций потом** (AI-генерация описаний). На GH Pages пришлось бы мигрировать. |
| Менеджер пакетов | bun | Скорость + встроенный test runner. |

### Структура репо (ориентир, не контракт)

```
src/
  pages/
    index.astro                       # лендинг с навигацией по системам
    mausritter/
      locations.astro                 # обёртка + MDX вступление
  components/
    LocationGenerator.tsx             # React-островок: кнопка, селектор, результат
    TableRoller/                      # будущий общий примитив
  data/
    mausritter/
      locations.ts                    # типизированные d20-таблицы
  layouts/
    ToolLayout.astro                  # общий шелл инструмента
  lib/
    dice.ts                           # roll(sides), pick(table)
```

URL-схема: `/<система>/<инструмент>` (например `/mausritter/locations`, позже `/mausritter/enemies`, `/cairn/...`).

## Деплой

Cloudflare Pages: подключаем репозиторий, билд-команда `bun run build`, выход `dist/`. На push в `main` — автоматический деплой, preview-деплои для PR из коробки.

Почему не GitHub Pages: серверная функциональность в будущем (AI-генерация описаний из выпавших таблиц) потребует прокси с секретами для API-ключа. На GH Pages этого сделать нельзя — пришлось бы либо мигрировать, либо клеить отдельный сервис. Cloudflare Pages даёт ту же бесплатную статику + Workers рядом, миграция не нужна.

Имя репозитория не зафиксировано. На Cloudflare Pages `base` в Astro-конфиге можно оставить `/` — деплой идёт на отдельный поддомен.

## Допущения и открытые вопросы

- **Лицензия Mausritter.** Известно, что Mausritter имеет открытый Third Party License (по сообществу — близко к CC BY 4.0), но это **не верифицировано прямо со страницы источника** в рамках брейнсторма. Перед публикацией контента — открыть [losing.games/mausritter](https://losing.games/mausritter/) и официальный SRD, выписать точный текст лицензии и требования к атрибуции. Если ограничения окажутся жёстче ожидаемого — пересмотреть формулировку футера и/или способ публикации перевода таблиц.
- **Имя репозитория.** Не зафиксировано. От него зависит `base` в Astro-конфиге.
- **«Только для себя» как обещание себе.** Если в какой-то момент захочется открыть платформу другим — потребуется ретроактивно докрутить i18n-инфраструктуру, мобильную адаптацию, SEO, осмысленную главную. Не строить это заранее, но помнить, что это будет вход в новую итерацию, а не «маленькая доделка».

## Не входит в продукт (в обозримом будущем)

- БД, аутентификация, синхронизация между устройствами.
- Многопользовательский режим, шаринг сессий.
- Поддержка любых систем кроме TTRPG (видеоигр, варгеймов и т. п.).
- Конструктор своих таблиц через UI — таблицы редактируются как код в репо.

## Перспективное расширение: AI-описания

Когда основа из 2-3 генераторов будет работать, появится отдельная итерация: брать выпавшие таблицы и собирать связное русскоязычное описание через LLM (Claude/локальная модель). Архитектурно это:

- Cloudflare Worker `/api/describe` — принимает JSON выпавших таблиц, возвращает текст.
- Секрет API-ключа хранится в переменных окружения Worker'а, не в клиенте.
- Клиент опционально вызывает эту функцию после броска, показывает loading и описание.

Эта возможность не делается в первой итерации, но выбор Cloudflare Pages сейчас оставляет эту дверь открытой без миграции хостинга.
